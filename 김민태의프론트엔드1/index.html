Chapter 02. Javascript 그리고 Typescript

  • Javascript 변천사.

    • ES2015부터가 모던 Javascript이고, ES5.0(EcmaScript5.0) 버전이 주도적인 버전이라고 볼 수 있음. 매년 새로운 버전이 출시되지만 모든 안정성을 위해 변환기(트랜스 파일러)를 통해 5.0 버전으로 변환하기 때문.
    • Typescript도 일종의 트랜스 파일러 중 하나임. JavaScript을 기본적으로 지원하고 거기에 +@되기 때문에 기본적으로 JavaScript를 기본적으로 알고 있어야 활용하기 좋다.

  • 웹앱의 구성요소.

    • 웹앱을 실행시키는 역할을 하는 것이 브라우저. 브라우저가 실행되는 시간을 런타임이라고 하고 브라우저를 런타임 환경을 제공하는 환경이라고 이야기한다. 웹앱에게 런타임 환경을 제공하는 것은 브라우저가 될 수 있다.
    • 예전에는 브라우저만 런타임 환경을 제공했지만, node.js가 등장해서 브라우저가 아니어도 JavaScript를 실행시킬 수 있는 환경이 다수 만들어짐.
    • 웹서버라는 소프트웨어가 HTML 파일을 브라우저한테 전송 - 브라우저가 HTML을 해석해서 화면에 UI를 그림 - JavaScript 실행. JavaScript도 HTML을 조작할 수 있기 때문에 브라우저가 웹앱을 실행시킨 다음에도 UI가 Javascript에 의해 조작되는 경우도 있다.
    • 클라이언트 사이드 렌더링 : 실제로 브라우저에 최초 전송된 HTML에는 거의 내용이 없고, JavaScript가 실행되면서 필요한 UI를 그때그때마다 생성해 내는 방식
    • 서버 사이드 렌더링: 웹서버에서 HTML이 만들어져서 브라우저로 전송되는 방식
    • 웹앱에서는 HTML과 CSS로 그래픽 시스템을 표현할 수 있고, JavaScript로는 캔버스라는 태그로 2D, 3D 등의 그래픽을 표현할 수 있다. JavaScript로 그래픽을 표현할 수 있도록 만들어진 도구들을 API라고 함.
    • 그 외에도 미디어 파일을 다루거나 웹 워커, 웹 어셈블리 등 다양한 스펙들이 존재한다.

  • 모던 JavaScript와 개발 환경.

  • 모던 JavaScript와 개발 환경.

    • ES2015부터 모던 JavaScript, 현대적인 자바스크립트라고 함.
    • 현대 프론트엔드 영역이 이렇게 복잡해진 가장 핵심적인 이유: 웹앱의 규모가 굉장히 커지고 복잡해졌다.
    • 자바스크립트는 애초에 HTML의 간단한 조작을 위해 만들어진 언어였다. 하나의 언어로 수만 개의 파일을 조작하려면 모듈이라는 스펙이 필요한데 JavaScript는 이 모듈이라는 게 없었다가 ES2015부터 지원하기 시작했다.
    • 모듈 스펙은 파일과 파일 간에 특정 파일의 어떤 기능을 사용하기 위해서 불러들여서 쓸 수 있는 기능으로 JavaScript에서는 import와 Export 문법으로 지원한다.
    • JavaScript가 실행되는 방식: HTML 안에 반드시 script태그를 이용해서 JavaScript 코드의 위치를 지정하게 되어 있다. JavaScript 파일을 로딩 즉 불러들이는 역할을 하는 태그. 문제는 자바스크립트 파일 수만큼 script태그를 넣어줘야 하는 문제가 생긴다. 이는 JavaScript 안에서 불러들이는 모듈 스펙도 아니고, HTML 안에 존재하면서 말 그대로 JavaScript파일을 불러들이기 때문에 수가 많이 늘어나면 굉장히 귀찮아진다. 그래서 JavaScript 코드 안에서 다른 JavaScript 코드를 불러들일 수 있는 import와 Export 문법이 생긴 것.
    • 그런데 import와 Export 문법을 그대로 사용하기 어려운 현실적인 문제 두 가지가 있음.
      1. 브라우저 호환성 문제(Can I Use)
      2. 업데이트되는 문법과의 호환성 문제
      그래서 엔지니어링을 시도하게 된다. node.js는 브라우저 대신 JavaScript를 실행시킬 수 있는 환경을 만들어줬고, npm에 개발자들이 다양한 툴들을 만들어 공유할 수 있게 되었다. 이러한 환경 변화 속에서 번들러라고 하는 소프트웨어가 탄생하게 됨.
    • 번들러는 기본적으로 시작되는 JavaScript 파일 하나를 지정하면 그 JavaScript 파일 안에서 모듈이라는 스펙을 통해 다른 JavaScript 파일을 불러들여 쓸 때, 이것들의 구문을 다 해석해서 하나의 파일로 만들어주는 역할을 한다. 브라우저에서 실제로 JavaScript 파일을 불러들이기 전에 하나의 파일로 만들어 놓으면, HTML 스크립트 태그에 그 하나의 파일만 연결해 놓으면 모든 게 해결된다.
    • 번들링이라는 형태에서 오는 문제점이 없진 않지만, 모듈 스펙을 못 쓰던 걸 쓸 수 있게 됐고, 많은 소스 코드를 가진 프로젝트라도 하나로 합칠 방법이 있으니까 HTML을 운영하기가 훨씬 편해졌다는 것이 큰 장점이다.
    • Webpack: 가장 많이 쓰는 번들러. JavaScript 파일 외에도 다양한 파일들을 하나로 합칠 수 있고, JavaScript, css, png 등의 다양한 아웃풋을 낼 수도 있다. 그리고 번들링 작업을 해놓고 보니 다양한 아이디어들이 나왔다.
      - 주석 제거: 용량을 줄이기 위해 주석들을 자동으로 제거
      - 이미지 최적화: 일정 용량 이상의 이미지 파일을 자동으로 압축하여 최적화
      - 어글리파이: JavaScript 중요 코드를 가려 보안성을 높일 수 있도록 코드를 못생기게 바꿔 가독성을 떨어지게 만듦
      다만 다양한 기능을 제공하다 보니 사용과 세팅 방법도 어려워지면서 번들러가 비대해지고 복잡해짐.
    • 제일 중요한 기능은 바로 트랜스 파일러
      - 인간이 이해할 수 있는 프로그래밍 언어를 기계가 이해할 수 있는 기계어로 변환시켜 줌. 최신 버전의 JavaScript 문법으로 개발을 하면 과거 버전으로 변환해주어 브라우저 호환성 문제를 해결해주는 것이 바로 babel임.
      - 번들러가 여러 파일을 하나로 합치고 있을 때 트랜스 파일러가 자연스럽게 과거 버전의 JavaScript 문법으로 바꾸는 일을 함. 그래서 요즘 번들러가 하는 중요한 역할 중 하나가 바로 트랜스 파일러를 끼워 넣어주는 것이다.
    • TypeScript
      실제로 JavaScript가 아니기 때문에 당연히 브라우저에서 실행될 수 없다. 그래서 TypeScript에서 제공하는 트랜스파일러를 통해 바로 원하는 JavaScript 버전으로 바꿀 수 있는 트랜스 파일링 동작이 일어난다. 그래서 TypeScript도 트랜스 파일러의 한 종류라고 할 수 있다.
    • 이처럼 번들링, 트랜스 파일링이라는 기능으로 변환을 할 수 있게 되면서 JavaScript의 한계를 JavaScript 뿐만 아니라 여러가지 아이디어들로 뛰어넘어 활용할 수 있게 되었다.
